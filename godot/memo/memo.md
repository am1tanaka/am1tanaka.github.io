# Godotの欠点

GDScriptの最大の欠点は、JavaScriptと同様の型定義のゆるさだと感じています。これにより大きなプロジェクトの開発は難しくなります。

実行するまで型が決まらない部分があるのでリファクタリング機能が弱くなります。Unityなどではメソッドや変数名の一括変更や参照先の検索がGodotではできません。またオートコンプリートも遅かったり表示されなかったりと弱いです。

ダックタイピングでポリモーフィズムを実現しているのも痛いです。C#などではメソッド名をタイプミスすれば親クラスやインターフェースに存在しないのでその場でエラーを出して教えてもらえます。GDScriuptはダックタイピングなので実行時に継承クラスをさかのぼって該当するメソッドを検索します。実行するまでタイプミスが見つからず、実行時にメソッドを検索するので動作も遅くなります。

エディターの機能不足も痛いです。UnityやUEでは、Visual StudioやRiderなどの活発な開発により瞬時にオートコンプリートで候補が表示されますし、最近ではAIがコード例を提案してくれます。GDScriptを編集するGodotの組み込みエディターはUnityで言えばリファクタリングができなかった時代までさかのぼるぐらいの機能しかありません。AIでの提案どころか、オートコンプリートも中途半端な状態です。

GodotのGDScriptはUnreal EngineのBlue Printと同じ扱いだという印象です。重要な部分はGDExtensionでC++やC#といった高速な言語で開発して、それをまとめるためにGDScriptを使うというスタンスのようです。

## 継承 extends

1行目は「`RigidBody3D`の機能を継承します。」という意味です。

継承とは、指定したものの機能を引き継いで利用したり、拡張する機能です。RigidBody3Dが提供する物理シミュレーションを使うためにRigidBody3Dを継承しています。

6行目の物理シミュレーションで速度などを設定するための`_integrate_forces(state)`を利用するために、RigidBody3Dを継承しています。

## 必要な情報の取得

以下の行では、処理に必要なGodotの情報を取得しています。

```python
@onready var viewport := get_viewport()
@onready var camera := get_viewport().get_camera_3d()
```

`@onready`は、スクリプトがアタッチされているノードが実行可能な状態になったら、後に続くスクリプトを実行する命令です。

`var`は、データを保存しておく<mark>変数</mark>を定義します。後ろに続く`viewport`と`camera`が変数名です。

`:=`は、<mark>型定義つきの代入</mark>です。代入する値と同時に、その型で変数が定義される便利な機能です。

`get_viewport()`は、ノードに割り当てられている[ビューポートを返すNodeクラスの関数](https://docs.godotengine.org/ja/4.x/classes/class_node.html#class-node-method-get-viewport)です。これが使えるのも、1行目の継承の効果です。RigidBody3DはNodeを継承しているので、Nodeの機能も使えるのです。マウスの座標を取得するために後ほど利用します。

`get_viewport()`は、Viewportクラスのインスタンスを返します。変数`viewport`はViewport型の変数として定義されて、このノードのビューポートが代入されます。

クラスとインスタンスについては[こちら](../class_and_instance/)にまとめました。ひとまずここではさらっと流して置いて構いません。

`get_camera_3d()`は、ビューポートに割り当てられている[カメラを返すViewportクラスの関数](https://docs.godotengine.org/ja/4.x/classes/class_viewport.html#class-viewport-method-get-camera-3d)です。ビューポートが必要なので、前に`get_viewport()`をつけています。`.`をつけることで、クラスの機能を呼び出すことができます。後ほど、マウスの座標をゲーム空間の座標に変換するのに使います。

## 物理シミュレーションへの介入

6行目の以下のは物理シミュレーションへ介入するための処理を書く関数の定義です。

```python
func _integrate_forces(state):
```

Godotは、キャラクターを制御するのに便利なクラスが２つ用意されています。一つは今回利用した[RigidBody3D](https://docs.godotengine.org/ja/4.x/classes/class_rigidbody3d.html)で、もう一つは[CharacterBody3D](https://docs.godotengine.org/ja/4.x/classes/class_characterbody3d.html#)です。

詳しくは公式マニュアルの[物理](https://docs.godotengine.org/ja/4.x/tutorials/physics/index.html)に書かれていますが、ざっくりいうと、RigidBody3Dは移動や衝突などを物理法則に基づいて自動的に制御するためのもの、CharacterBody3Dは当たり判定などは物理的な性質を利用しつつ、移動は直接制御するためのものという感じです。弾などの物理法則に基づいて自動的に移動するものはRigidBody3D、歩行したり斜面を登ったりジャンプしたりするキャラクターはCharacterBody3Dを使うのが一般的です。

今回は動きとしてはCharacterBody3Dの方が性質は近いのですが、CharacterBody3Dでは衝突したRigidBodyに自動的に力を加えることができません。今回は球体を操作してカプセルを持ち上げたり、物理的な反応をさせたいので、RigidBody3Dを利用しました。

UnityやUnreal Engineに比べて優れているところは以下のところです。

- 完全なオープンソースプロジェクト
  - プレイヤー情報を勝手に引っこ抜かれることがない(ランタイムフィー事件のようなことは絶対におきない)
  - 自分でメンテさえすれば、バージョンを自由に選べる
- 性能の低いハードウェアに寄り添っている
  - シンプルな実装になっていて、古の技術を使えばメモリやビデオの性能が相当低くても動くものが作れる

正直、この2点ぐらいかも^^; 劣っているところは数え上げたらきりがないほどあります。現実的なことを言えば、開発規模が全く違うので別物と考えた方がいいです。Unreal EngineやUnityがバンダイのガンプラだとすれば、Godotは有志が出しているガレージキットです。マニュアルを見れば凄いのができあがるPG UNLEASHEDの感覚で取り組むと、バリ取りやら塗装やら場合によっては自分でパーツの付け足しが必要になるガレキ






習得難易度はUnreal EngineやUnityより<mark>やや難しい</mark>という印象です。


また公式な推奨スクリプトのGDScriptの効率が良くなく、またダックタイピングを使うため、JavaScriptと同様の実行時にならないとタイプミスのエラーを発見できない、という問題を抱えています。リファクタリングも弱く、クラス名や関数名、変数名の一括変更ができないので、大きいプロジェクトで使うのは難しいと感じます。
